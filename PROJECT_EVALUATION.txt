SECTION 5: EXAMINER QUESTIONS & MODEL ANSWERS
================================================================================

──────────────────────────────────────────────────────────────────────

Q1: "What problem does this project solve?"

A: "Most people write AI prompts by trial and error — type something,
    get a bad result, try again. There's no structured way to LEARN
    prompt engineering, no tool to BUILD prompts systematically, and
    no easy way to TEST across models. Blog posts don't teach by doing.
    ChatGPT itself doesn't teach you how to use it better.

    We built a platform that combines education, tooling, and testing
    into one place. You learn the fundamentals through interactive
    exercises, build prompts using a structured form, test them in a
    playground, and compare outputs across models — all without leaving
    the app."

──────────────────────────────────────────────────────────────────────

Q2: "Walk me through the tech stack and architecture."

A: "This is a Next.js 16 application using the App Router with React 19
    and TypeScript. The architecture follows a clean separation:

    - /app contains all route pages with per-page SEO layout files
    - /components/ui has reusable components (Navbar, CopyButton, etc.)
    - /lib has custom hooks (useTheme, useLocalStorage, useProgress)
      and business logic (exercises, promptHelpers)
    - /styles has the CSS design system (variables, typography, layout)

    We use Framer Motion for animations, Lucide for icons, and Three.js
    for the 3D particle wave on the landing page. State is managed with
    React hooks + localStorage for persistence. The design system uses
    CSS custom properties for theming."

──────────────────────────────────────────────────────────────────────

Q3: "How does the tokenizer work technically?"

A: "The tokenizer uses a regex-based approximation that splits text on
    word boundaries, contractions, and common subword patterns. It's not
    the exact BPE tokenizer that GPT uses (that would require the tiktoken
    library), but it demonstrates the concept accurately.

    The BPE visualization is a step-by-step implementation: it starts
    with individual characters, counts byte-pair frequencies, merges the
    most common pair, and repeats. Users can step through each merge to
    understand how 'unbelievable' becomes ['un', 'believ', 'able'].

    The cost calculator uses real pricing data from OpenAI, Anthropic,
    and Google to show how token count translates to actual API costs."

──────────────────────────────────────────────────────────────────────

Q4: "The AI API is mocked. How would you integrate real APIs?"

A: "The architecture is already built for it. The app/api/generate/route.ts
    file currently calls a mockAIResponse function. To integrate real APIs:

    1. Install the OpenAI SDK: npm install openai
    2. Create a .env.local file with OPENAI_API_KEY
    3. Replace mockAIResponse() with:
       const openai = new OpenAI();
       const response = await openai.chat.completions.create({
         model: model,
         messages: [{ role: 'user', content: prompt }],
         temperature: temperature,
         max_tokens: maxTokens,
       });

    We kept it mocked intentionally so the demo works without API keys,
    and there's no risk of exposing keys or incurring costs during the
    competition. The mock data demonstrates the full UX flow faithfully."

──────────────────────────────────────────────────────────────────────

Q5: "How does the prompt quality scoring work?"

A: "The scorePrompt function in lib/promptHelpers.ts analyzes prompt
    structure, not content. It checks for the presence of:

    - Role definition (e.g., 'You are a...')
    - Clear task instruction
    - Context or background information
    - Output format specification
    - Constraints (word limits, rules)
    - Examples (few-shot patterns)
    - Tone specification

    Each element adds points. A well-structured prompt with all elements
    scores 80-100. A single-sentence prompt scores 10-20. This teaches
    users that prompt quality is about STRUCTURE, not just wording."

──────────────────────────────────────────────────────────────────────

Q6: "How do you handle data persistence without a backend?"

A: "We use the browser's localStorage API through a custom React hook
    called useSavedPrompts. It provides CRUD operations (save, update,
    delete), tag-based filtering, and search.

    We also built export/import functionality — users can download their
    entire prompt library as a JSON file and import it on another device.
    This partially solves the cross-device limitation without needing a
    backend.

    For the learning progress, we have a separate useProgress hook that
    tracks which modules are completed. Both hooks serialize to JSON
    and use the 'peh-' prefix to avoid key collisions.

    The next evolution would be Firebase Auth + Firestore for cloud
    persistence, which we've documented in the README as a roadmap item."

──────────────────────────────────────────────────────────────────────

Q7: "What's the 3D animation on the homepage? Why Three.js?"

A: "The landing page uses a Three.js particle system — 2,400 dots
    arranged in a 40×60 grid, animated with sine waves to create a
    flowing surface effect. It adapts to the theme: light gray dots in
    dark mode, black dots in light mode.

    We chose Three.js because it's the industry standard for WebGL in
    React applications. The component is self-contained, handles cleanup
    properly (disposing geometries, materials, and the WebGL context),
    and responds to window resizing. It runs at 60fps with minimal
    CPU usage because it's GPU-accelerated."

──────────────────────────────────────────────────────────────────────

Q8: "How did you handle dark mode across the entire app?"

A: "Dark mode uses CSS custom properties (CSS variables). We define a
    complete color palette in styles/variables.css under a
    [data-theme='dark'] selector. When the theme changes, we update the
    data-theme attribute on the <html> element.

    The ThemeProvider (a React context) manages the state, reads the
    user's system preference via matchMedia, and persists the choice to
    localStorage. Every CSS component references variables like
    --bg-primary, --text-primary, --border, so switching the data-theme
    attribute instantly recolors the entire app.

    For Three.js, the DottedSurface component re-renders when the theme
    changes, creating the particles with the appropriate color values."

──────────────────────────────────────────────────────────────────────

Q9: "What accessibility features did you implement?"

A: "We focused on four areas:

    1. ARIA attributes: role='navigation' on the navbar, role='main'
       on the content area, aria-current='page' on active nav links,
       aria-label on all icon-only buttons.

    2. Keyboard navigation: A global focus-visible style ensures all
       interactive elements show a clear 2px outline when focused
       via keyboard.

    3. Live regions: The playground output area has aria-live='polite'
       so screen readers announce new output without interrupting.

    4. Semantic HTML: Proper heading hierarchy (one h1 per page),
       list roles on the dashboard prompt list, and semantic
       landmarks throughout.

    Areas for improvement include a skip-to-content link, full
    WCAG 2.1 AA contrast audit, and screen reader testing."

──────────────────────────────────────────────────────────────────────

Q10: "If you had two more weeks, what would you add?"

A: "Three things:

    1. Real API integration — the single biggest impact. Connect OpenAI's
       API so the playground gives real responses.

    2. User accounts with Firebase Auth — so progress and prompts sync
       across devices. The localStorage approach works for a demo but
       isn't production-ready.

    3. A 'Prompt Diff' feature — when you iterate on a prompt, show a
       visual diff of what changed and how the output improved. This
       would be a genuinely novel feature that no other platform offers."

──────────────────────────────────────────────────────────────────────

Q11: "What's your competitive landscape?"

A: "There are three categories of competitors:

    AI Learning Platforms (Coursera, DeepLearning.AI):
    → They teach theory in video format. We teach through interactive
      exercises with instant feedback.

    Prompt Marketplaces (PromptBase, PromptHero):
    → They sell pre-built prompts. We teach you to BUILD your own
      and understand WHY they work.

    Prompt Tools (PromptPerfect, Agenta):
    → They optimize existing prompts automatically. We give you the
      UNDERSTANDING to optimize yourself.

    Nobody combines education + building + testing in one interactive
    platform. That's our niche."

──────────────────────────────────────────────────────────────────────

Q12: "How scalable is this application?"

A: "Currently it's a client-heavy application with localStorage — it
    scales infinitely because there's no server load per user. The only
    server interaction is the /api/generate endpoint.

    For production scaling:
    - The API route can be deployed on Vercel Edge Functions (global CDN)
    - User data would move to Firestore (auto-scales)
    - Static pages (Learn, Patterns) could be ISR (Incremental Static
      Regeneration) for instant loading
    - The Three.js component lazy-loads (only homepage payload)

    The architecture is ready for production — it just needs the backend
    services connected."
