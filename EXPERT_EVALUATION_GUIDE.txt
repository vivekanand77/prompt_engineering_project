2. DOES THE STUDENT UNDERSTAND ATTENTION MECHANISMS? [Weight: 10%]
──────────────────────────────────────────────────────────────────────────

What I expect:
  - Know that transformers process ALL tokens simultaneously (not L→R)
  - Understand that self-attention computes relevance between every
    pair of tokens using Query, Key, Value matrices
  - Know that multi-head attention allows the model to focus on
    different aspects of the input simultaneously
  - Understand that this is WHY word order matters in prompts

What this project demonstrates:
  ✅ Attention heatmap visualization in the Simulation page
  ✅ Interactive — users see how tokens relate to each other

What the student should SAY:
  "The transformer architecture doesn't read left-to-right like humans.
   It uses self-attention: for every token in the input, it computes
   an attention score against every other token. This is calculated as:

   Attention(Q,K,V) = softmax(QK^T / √d_k) × V

   Where Q, K, V are learned projection matrices. The softmax
   normalizes the scores so they sum to 1, creating a probability
   distribution of 'how much should I pay attention to each other
   token when processing this one.'

   In our visualization, each cell in the heatmap shows this attention
   weight. Darker cells mean stronger attention — meaning those two
   tokens are more relevant to each other.

   This is why prompt engineering works: by changing the order and
   structure of words, we directly influence these attention patterns,
   guiding the model to focus on what matters."

3. DOES THE STUDENT UNDERSTAND GENERATION / DECODING? [Weight: 10%]
──────────────────────────────────────────────────────────────────────────

What I expect:
  - Know that LLMs are autoregressive — they predict ONE token at a time
  - Understand temperature: T=0 (greedy/deterministic) vs T=1 (diverse)
  - Know what top-p (nucleus sampling) does
  - Understand that the model outputs a probability distribution over its
    entire vocabulary (~100K tokens) for each next-token prediction

What this project demonstrates:
  ✅ Generation simulation — token-by-token output with animation
  ✅ Deterministic vs Creative toggle (temperature concept)
  ✅ Temperature slider in the Playground

What the student should SAY:
  "Language models don't generate text — they generate probability
   distributions. For each position, the model outputs a vector of
   ~100,000 probabilities, one for each token in its vocabulary.

   Temperature controls how 'peaked' this distribution is:
   - T=0: Always picks the highest-probability token (greedy decoding)
   - T=0.7: Slightly random, balancing quality and diversity
   - T=1.0: Full softmax distribution, more creative but less reliable
   - T>1.0: Flattens the distribution, making rare tokens more likely

   Mathematically: P(token_i) = exp(logit_i / T) / Σ exp(logit_j / T)

   In our Playground, you can set the temperature before running a
   prompt and observe how the same prompt produces different outputs
   at different temperatures. The Simulation page shows this process
   token-by-token."

What the student should SAY:
  "A prompt is an engineering artifact, not a sentence. Professional
   prompt engineers decompose prompts into components:

   1. SYSTEM PROMPT — Sets the model's persona and constraints.
      This works because the system message occupies early positions
      in the attention window, establishing strong attention patterns
      that influence every subsequent token.

   2. ROLE DEFINITION — 'You are a senior data scientist' triggers
      the model to activate related knowledge clusters. This works
      because during pre-training, content about data science was
      often preceded by author credentials, creating a learned
      association between role context and expertise-level output.

   3. FEW-SHOT EXAMPLES — Providing 2-3 input/output examples
      works because of in-context learning: the model's attention
      mechanism identifies patterns across examples and applies
      them to the new input. This is emergent behavior — it wasn't
      explicitly trained, it arises from the architecture.

   4. CHAIN OF THOUGHT — Asking the model to 'think step by step'
      works because it forces the model to generate intermediate
      reasoning tokens. These tokens then appear in the attention
      window for subsequent tokens, creating a chain of logical
      dependencies. Without CoT, the model must compute the answer
      in a single forward pass — for complex problems, that's
      insufficient.

   5. OUTPUT FORMAT — Specifying 'respond in JSON' or 'use bullet
      points' constrains the generation space, reducing the
      probability mass allocated to wrong-format tokens.

   Our Builder enforces this structure. The scoring algorithm
   checks for the presence of each component and rates the prompt
   0-100 based on structural completeness."

Q1: "Why does Chain of Thought prompting work at a mechanistic level?"

A: "CoT works because LLMs are autoregressive — each token is generated
    conditioned on all previous tokens. When you force the model to
    generate intermediate reasoning steps, those step-tokens become part
    of the context window for subsequent predictions.

    For example, to solve 'If A=3 and B=A+2, what is B²?':
    - Without CoT: The model must compute A=3, B=5, B²=25 in a SINGLE
      forward pass through 96 transformer layers. This is possible for
      simple arithmetic but fails for multi-step reasoning because the
      model's hidden state doesn't have enough 'compute steps.'
    - With CoT: The model first generates 'A=3' (now visible in context),
      then 'B=3+2=5' (using A from context), then 'B²=5²=25' (using B
      from context). Each step is a separate forward pass with the
      previous result in the attention window.

    CoT effectively gives the model more 'serial computation depth' by
    converting parallel internal computation into sequential token
    generation. This is related to the concept of 'chain-of-thought
    faithfulness' — whether the model's generated reasoning actually
    reflects its internal computation."

──────────────────────────────────────────────────────────────────────────

Q2: "What's the difference between fine-tuning and prompt engineering?
     When would you choose one over the other?"

A: "They solve different problems on the adaptation spectrum:

    PROMPT ENGINEERING:
    - Zero training cost, instant iteration
    - Works with any API (no model access needed)
    - Limited by context window size
    - Per-request latency is higher (longer prompts = more tokens)
    - Best for: variable tasks, rapid prototyping, when you need
      to change behavior frequently

    FINE-TUNING:
    - Requires training data (hundreds to thousands of examples)
    - Training costs $5-100+ depending on model size
    - Results in a specialized model with shorter prompts needed
    - Lower per-request latency and cost (don't need long prompts)
    - Best for: repetitive tasks, brand voice, domain-specific
      formatting, when consistency matters

    In practice, the best approach is: start with prompt engineering,
    measure quality, then fine-tune if quality plateaus and the task
    is high-volume. A well-engineered prompt on GPT-4 often matches
    a fine-tuned GPT-3.5 — it's a cost/quality tradeoff.

    Our platform focuses on prompt engineering because it's the
    universal skill — you need it regardless of whether you eventually
    fine-tune."

──────────────────────────────────────────────────────────────────────────

Q3: "Explain the hybrid data persistence architecture."

A: "We use a three-layer strategy:

    Layer 1: LOCAL STATE (React useState/useEffect)
    → Immediate UI responsiveness, zero latency

    Layer 2: LOCAL STORAGE (browser)
    → Persistence across page reloads, works offline
    → Used when user is NOT signed in

    Layer 3: FIRESTORE (cloud)
    → Cross-device sync, data backup
    → Used when user IS signed in

    The hybrid hooks (useSyncedPrompts, useSyncedProgress) abstract
    this completely. They check useAuth().user:
    - If user exists → read/write Firestore
    - If user is null → read/write localStorage

    On first sign-in, migrateLocalDataToFirestore() runs:
    1. Reads all localStorage data
    2. Batch-writes it to Firestore (using writeBatch for atomicity)
    3. Clears localStorage to avoid data duplication

    This means the app has zero configuration burden on the user:
    - Works immediately without sign-in (localStorage)
    - Upgrades seamlessly when they sign in (Firestore)
    - Existing data is never lost during the transition"

Q4: "How does the prompt scoring algorithm work, and what are its
     limitations?"

A: "The scorePrompt function in lib/promptHelpers.ts performs STRUCTURAL
    analysis, not semantic analysis. It checks for:

    Component         Points    Detection Method
    ─────────────────────────────────────────────
    Role definition    +15      Regex: 'you are', 'act as'
    Clear instruction  +20      Length > 20 chars, imperative verbs
    Context            +15      Keywords: 'context', 'background'
    Output format      +15      Keywords: 'format', 'json', 'list'
    Constraints        +10      Keywords: 'must', 'don't', 'max'
    Examples           +15      Pattern: 'example', 'input:', 'output:'
    Tone specification +10      Keywords: 'tone', 'formal', 'casual'
    ─────────────────────────────────────────────
    Maximum            100

    LIMITATIONS:
    1. It's syntactic, not semantic — 'be professional' scores points
       for tone even if the rest of the prompt is nonsensical
    2. No evaluation of prompt EFFECTIVENESS — just structure
    3. Regex-based detection can miss creative phrasing
    4. Doesn't consider model-specific best practices

    A real production scorer would use an LLM-as-judge approach:
    send the prompt to GPT-4 with a meta-prompt asking it to rate
    clarity, specificity, and structural completeness. That requires
    API access, which is our next milestone."

──────────────────────────────────────────────────────────────────────────

Q5: "What happens when you exceed a model's context window?"

A: "Context window overflow is one of the most common production failures.

    When the total tokens (system prompt + user prompt + conversation
    history + expected output) exceed the model's window:
    - GPT-4 Turbo: 128K tokens
    - Claude 3: 200K tokens
    - Gemini Pro: 32K tokens

    The API returns an error. The prompt is NOT silently truncated
    (unlike some local models that use a sliding window).

    Mitigation strategies:
    1. PROMPT OPTIMIZATION — reduce token count while preserving meaning
       (our Learn module 5 teaches this)
    2. RAG (Retrieval Augmented Generation) — instead of stuffing
       everything into the prompt, retrieve only relevant chunks
    3. SUMMARIZATION CHAINS — summarize long inputs before prompting
    4. PROMPT CHAINING — break complex tasks into smaller prompts
       (our Learn module 8 covers this)

    Our tokenizer visualization helps users monitor token counts
    before they hit API limits."

──────────────────────────────────────────────────────────────────────────

Q6: "Why did you use CSS custom properties instead of Tailwind for
     theming, and how does your dark mode work under the hood?"

A: "CSS custom properties (variables) are a better primitive for theming
    than Tailwind's class-based approach because they allow RUNTIME
    changes without recomputation.

    The dark mode works in four layers:

    1. CSS VARIABLES defined in styles/variables.css:
       :root { --bg-primary: #FAFAFA; --text-primary: #0A0A0A; }
       [data-theme='dark'] { --bg-primary: #0A0A0A; --text-primary: #F5F5F5; }

    2. REACT CONTEXT (ThemeProvider) manages the state:
       - Reads system preference via matchMedia
       - Reads saved preference from localStorage
       - Updates document.documentElement.setAttribute('data-theme', theme)

    3. ALL COMPONENTS reference variables:
       .btn { color: var(--text-primary); border: 2px solid var(--border); }
       → No component needs to know about dark mode. The variables handle it.

    4. THREE.JS INTEGRATION — DottedSurface recreates particles when
       theme changes, using light-gray dots (200,200,200) for dark mode
       and black dots (0,0,0) for light mode.

    The toggle is instant because changing data-theme triggers a CSS
    recomputation only for properties that reference the changed
    variables — no JavaScript DOM manipulation needed."

PART 4: KEY KNOWLEDGE SUMMARY — WHAT TO MEMORIZE BEFORE THE EXPO
================================================================================

CONCEPT                 ONE-LINE EXPLANATION
─────────────────────   ──────────────────────────────────────────────────
Token                   Subword unit; smallest piece of text an LLM processes
BPE                     Algorithm that builds vocabulary by merging frequent byte pairs
Context Window          Maximum tokens a model can process in one request
Temperature             Controls randomness: 0=deterministic, 1=creative
Top-P                   Nucleus sampling: only consider tokens in top P% probability
Attention               Mechanism computing relevance between all token pairs
Self-Attention          Attention applied within the same sequence (not cross-sequence)
Multi-Head Attention    Running multiple attention computations in parallel
Autoregressive          Generating tokens one at a time, left to right
System Prompt           Hidden instruction that persists across conversation turns
Few-Shot                Providing examples in the prompt for in-context learning
Chain of Thought        Asking the model to show reasoning steps before answering
ReAct                   Combining reasoning with external tool use (Reason + Act)
Role Prompting          Setting a persona to activate relevant knowledge clusters
Guardrails              Constraints that prevent unwanted model behavior
Fine-Tuning             Training a model on custom data (vs prompt engineering)
RAG                     Retrieving relevant documents to augment the prompt
Embedding               Vector representation of text for similarity search
Hallucination           Model generating confident but factually wrong content
Prompt Injection        Adversarial input that overrides the system prompt
